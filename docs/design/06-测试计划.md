## 5. 缺陷管理

### 5.1 缺陷分级

| 级别 | 定义 | 响应时间 | 修复时间 |
|------|------|----------|----------|
| P0-致命 | 系统崩溃、数据丢失、核心功能不可用 | 1小时 | 4小时 |
| P1-严重 | 重要功能失效、数据错误、安全问题 | 4小时 | 24小时 |
| P2-一般 | 功能缺陷、交互问题、性能问题 | 1天 | 3天 |
| P3-轻微 | UI问题、建议改进 | 2天 | 1周 |

### 5.2 缺陷状态流转

```
新建 → 确认 → 分配 → 修复 → 验证 → 关闭
            ↓
          延期 ← 拒绝
```

| 状态 | 说明 | 操作人 |
|------|------|--------|
| 新建(New) | 缺陷提交 | 测试工程师 |
| 确认(Confirmed) | 缺陷确认有效 | 开发组长 |
| 分配(Assigned) | 分配给开发人员 | 开发组长 |
| 修复(Fixed) | 开发完成修复 | 开发工程师 |
| 验证(Verified) | 测试验证通过 | 测试工程师 |
| 关闭(Closed) | 缺陷关闭 | 测试组长 |
| 延期(Deferred) | 延期处理 | 项目经理 |
| 拒绝(Rejected) | 不是缺陷 | 开发组长 |

### 5.3 缺陷提交规范

```markdown
缺陷编号: BUG-XXX-001
缺陷标题: [模块] XXX问题描述
严重级别: P0/P1/P2/P3
优先级: 高/中/低
环境: 测试环境
浏览器: Chrome 120
操作系统: Windows 11
版本: v1.0.0

【缺陷描述】
详细描述缺陷的表现

【复现步骤】
1. 进入XXX页面
2. 点击XXX按钮
3. 输入XXX数据
4. 点击保存

【预期结果】
应该如何表现

【实际结果】
实际表现是什么

【附件】
截图、日志等
```

### 5.4 缺陷分析指标

| 指标 | 目标 | 计算方法 |
|------|------|----------|
| 缺陷总数 | - | 提交的缺陷数量 |
| 致命/严重缺陷占比 | ≤20% | P0+P1/总缺陷 |
| 缺陷修复率 | ≥95% | 已修复/总缺陷 |
| 缺陷reopen率 | ≤5% | 被reopen/已修复 |
| 平均修复时间 | ≤24小时 | 总修复时间/缺陷数 |
