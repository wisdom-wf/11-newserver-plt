# 业务逻辑偏差报告

> 整理：董老师 | 日期：2026-05-15
> 基于代码审查 + 数据库现状 + 角色体系梳理

---

## 一、目标业务流程（你期望的）

```
预约确认
  → 自动生成订单(CREATED)
  → 平台/政府给服务商派单(DISPATCHED)
  → 护工接单(RECEIVED)
  → 护工上门服务（服务中）
  → 护工提交服务日志（触发质检）
  → 质检员质检
    ├── 合格 → 订单完成(COMPLETED)
    └── 不合格 → 整改 → 复检
  → 订单完成 → 服务商发起满意度评价
  → 老人/家属填问卷(EVALUATED)
  → 结算(SETTLED)
```

**数据沉淀**：每个环节的数据（老人档案、服务记录、体征、评价、质检）全部汇聚到**健康档案模块**，形成数据资产。

## 二、现状：核心实体状态机

### 2.1 订单状态机（t_order.status）

| 状态码 | 名称 | 说明 |
|--------|------|------|
| CREATED | 待派单 | 预约确认后自动创建 |
| DISPATCHED | 已派单 | 派单给护工 |
| RECEIVED | 已接单 | 护工确认接单 |
| SERVICE_STARTED | 服务中 | 存在但无触发入口 |
| SERVICE_COMPLETED | 已完成 | 中间状态，未被使用 |
| COMPLETED | 已完成 | 质检合格后联动设置 |
| EVALUATED | 已评价 | 评价提交后 |
| SETTLED | 已结算 | 财务结算 |
| REJECTED | 已拒单 | 护工拒单 |
| CANCELLED | 已取消 | 管理员取消 |

### 2.2 服务日志审核流（service_log.audit_status）

```
DRAFT(草稿)
  → [护工提交] → SUBMITTED(已提交)
      → [质检合格/审核通过] → APPROVED(通过)
      → [质检不合格/审核拒绝] → REJECTED(拒绝) + 可整改
      → [复检合格] → COMPLETED(完成)
```

### 2.3 质检流（quality_check）

```
PENDING(待质检)
  → [质检员结论: QUALIFIED] → 订单→COMPLETED + 日志→APPROVED
  → [质检员结论: UNQUALIFIED/NEED_RECTIFY] → 整改流程(RECTIFY)
      → [整改提交] → RECHECK
      → [复检PASSED] → 订单→COMPLETED + 日志→COMPLETED
      → [复检FAILED] → 退回PENDING
```

## 三、发现的偏差（共6个核心问题）

### 【偏差-1】护工（STAFF）没有独立 App 操作入口

**期望**：护工用自己的手机登录 App → 查看待接单 → 接单 → 完成服务后提交服务日志

**现状**：
- `t_user` 表中有一个 `user_type=STAFF` 的账号（手机号 13109118901），但这是**护工手机号注册登录**产生的账户
- `PermissionInterceptor` 让 STAFF 用户可以 `GET` 所有接口，但**写操作（接单、提交日志）实际上是通过 PROVIDER_ADMIN 账号完成的**
- 护工信息（`t_staff` 表）是独立存在的，但**护工没有独立系统账号**，无法自己登录系统操作
- 订单派单后，护工看到的"接单"操作实际由 PROVIDER_ADMIN 代为操作

**影响**：系统无法追踪哪个护工实际执行了服务，服务日志中的 `staff_id` 是派单时指定的，但护工本人无法主动确认"我到了、我完成了"

**建议**：
1. 给每个 `t_staff` 记录生成独立登录账号（手机号即用户名 + 默认密码）
2. 开发**护工 App 端**（或微信小程序）：接单、开始服务、提交服务日志、查看任务
3. 护工提交服务日志时，校验当前登录账号的 `staff_id` 与派单记录中的 `staff_id` 一致

---

### 【偏差-2】没有"开始服务"节点，订单状态无法进入 SERVICE_STARTED

**期望**：护工到达老人家门口 → 点"开始服务" → 订单进入 SERVICE_STARTED（服务中）

**现状**：
- `OrderStatus.SERVICE_STARTED` 枚举值存在，数字码为 5
- `OrderServiceImpl` 中**没有任何方法将订单置为 SERVICE_STARTED**
- 订单从 RECEIVED（已接单）直接跳到 COMPLETED（质检合格后）
- 没有"开始服务"的 API 端点

**影响**：
1. 平台无法监控护工是否真正开始上门服务（只有接单记录，没有上门记录）
2. 护工可以跳过"到岗"环节直接提交日志
3. 缺少到岗打卡（sign_in_time / sign_in_location）的人为节点校验

**建议**：
1. 新增 `POST /api/service-log/start` 接口（护工点击"开始服务"）：
   - 校验：订单状态 = RECEIVED，当前护工 staff_id = 派单记录中的 staff_id
   - 操作：订单 → SERVICE_STARTED，记录 sign_in_time + sign_in_location
2. 提交服务日志时，强校验 sign_in_time 存在（即必须先开始服务才能提交）

---

### 【偏差-3】服务日志提交即触发质检，但质检角色不清

**期望**：
- 护工提交日志 → 服务商质检员审核日志质量
- 质检员可以是：政府监管方 / 平台质控 / 服务商内部质控

**现状**：
- `ServiceLogServiceImpl.submitServiceLog()` 在提交时**自动创建 `quality_check` 记录**（check_result=PENDING）
- `QualityCheckServiceImpl.inspect()` 是质检员操作，但：
  - `QualityCheckServiceImpl` 中**没有任何角色限制**，只要能调接口就能质检
  - `checker_id` 是从 `UserContext` 取的，但没有任何校验说"只有某角色才能质检"
- `PermissionInterceptor` 放行了 PROVIDER 的 `/api/quality-check` 写操作，所以 PROVIDER_ADMIN 可以质检

**影响**：
1. 质检角色混乱——谁都能质检，没有专职质检员概念
2. 平台监管方（政府）无法介入质检流程
3. 服务商自己给自己质检，缺乏第三方监督

**建议**：
1. 新增**平台质控角色**（PLATFORM_QC_ADMIN）：可以质检所有服务商的订单
2. PROVIDER_ADMIN 只能质检自己公司的订单（加 `providerId` 校验）
3. 质检结论触发：
   - `QUALIFIED` → 订单→COMPLETED + 日志→APPROVED（已有）
   - `UNQUALIFIED` → 护工收到整改通知，整改后重新提交日志
4. 考虑引入**政府监管角色**（GOVERNMENT_QC）：可抽检任订单

---

### 【偏差-4】满意度评价未嵌入业务流程节点

**期望**：订单 COMPLETED 后，**系统自动**向老人/家属发送满意度评价邀请，老人填完问卷后订单才进入 EVALUATED

**现状**：
- `EvaluationController.generateEvaluationLink()` 需要**手动调用**生成评价链接
- 没有在订单 COMPLETED 时自动触发
- 老人通过 token 链接填写问卷，token 有时效（默认72小时）
- `t_service_evaluation.token_status=PENDING` → 老人填完后 `COMPLETED`
- 但**订单不会自动变成 EVALUATED**——`OrderServiceImpl` 和 `QualityCheckServiceImpl` 中都没有更新订单为 EVALUATED 的逻辑

**影响**：
1. 评价环节游离在业务流程之外，变成了"可选步骤"
2. 大量订单永远停在 COMPLETED，不会进入 EVALUATED
3. SETTLED（结算）不依赖 EVALUATED，导致可以在未评价时就结算

**建议**：
1. `QualityCheckService.inspect()` 合格后，**自动调用** `EvaluationService.generateEvaluationLink(orderId, elderId, elderName)`
2. 老人提交问卷后，`submitSurveyByToken()` 中**联动更新 `order.status=EVALUATED`**
3. 结算前置条件：订单状态必须 ≥ EVALUATED

---

### 【偏差-5】健康档案模块未与业务链路打通

**你对这个模块的定位**：数据资产沉淀中心，所有业务流程中的数据最终汇聚到这里

**现状**：
- 健康档案（`t_elder_health` / `t_health_measurement` / `t_health_report`）是独立模块
- **没有任何一个业务环节主动向健康档案写入数据**
- 服务日志中有 `health_observations`（健康观察）字段，但只是文字记录，没有写入健康档案
- 体征数据（血压、心率等）没有与订单/服务日志建立关联

**建议**：
1. **服务日志提交时**，将 `health_observations` 和 `medication_given` 写入 `t_health_measurement`
2. **护工 App 端"测体征"功能**，测完后直接写入 `t_health_measurement`（设备绑定场景）
3. **质检完成后**，将质检结论（合格率、评分）写入 `t_health_report`
4. **满意度评价完成时**，将 `overall_score` 写入 `t_health_report`
5. **健康档案页面**按老人 ID 聚合展示：所有服务记录 + 所有质检结论 + 所有评价得分 + 所有体征数据

---

### 【偏差-6】评价发起角色设计不合理（谁发起的？）

**现状**：
- `generateEvaluationLink()` 可以由任何能调用该接口的人调用
- PROVIDER_ADMIN 有权限，SYSTEM_ADMIN 也有权限
- **没有规定"只能由服务商在订单完成后发起评价"**

**你期望的**：服务商在订单界面看到已完成订单 → 点击"发起满意度评价"

**建议**：
1. 限制 `generateEvaluationLink()` 只能由 PROVIDER_ADMIN 调用（加 `userType=PROVIDER` 校验）
2. 前端：已完成订单列表增加"发起评价"按钮（仅 PROVIDER_ADMIN 可见）
3. 同一个订单只能生成一个有效 token，作废旧 token 再生成新 token
