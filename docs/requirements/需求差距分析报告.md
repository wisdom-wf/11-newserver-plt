## 二、服务商入驻管理模块 — 差距分析

### 2.1 已有功能（✅）

| 功能 | 实现位置 |
|------|---------|
| 营业执照管理 | `t_provider.businessLicense` 字段（Base64存储） |
| 健康证管理 | `t_staff_qualification.qualificationType=1` |
| 服务类型配置 | `t_provider_service_type` 表 + ProviderController 接口 |
| 服务评分展示 | `t_provider_score` 表 + ProviderScoreController |
| 服务区域管理 | `Provider.serviceAreas` + `PUT /api/providers/{id}/service-areas` |
| 账户管理 | 注册时自动创建管理员账号；密码重置 |

**已有实体**：`t_provider`, `t_provider_qualification`, `t_provider_service_type`, `t_provider_score`（4张表）

### 2.2 缺失功能（❌）

| 功能 | 需求描述 | 当前状态 | 影响 |
|------|---------|---------|------|
| **审核流程** | 资质需区县管理员审核（待审核→通过/拒绝→补充材料） | 无 auditStatus 字段，无审核接口 | 服务商无法正式入驻 |
| **信用等级** | 信用等级(AAA/AA/A/B/C) + 动态信用分计算 | 无 credit_level / credit_score 字段 | 无法淘汰低分服务商 |
| **绩效考核** | 月度/季度/年度考核 + 5维度评分 + 考核报告 | 无 t_provider_assessment 实体 | 无法量化服务商绩效 |
| **退出管理** | 退出流程（停止派单→处理订单→资金结算→归档） | 无退出管理接口 | 服务商无法正规退出 |
| **经营信息** | 法人电话、注册地址、经营地址、经营范围 | 仅有 address / legalPerson 两个字段 | 信息不完整 |
| **财务信息** | 银行账号、开户行、税号 | 无对应字段 | 无法做薪资结算 |
| **批量审核** | 支持批量审核操作 | 无批量接口 | 效率低 |
| **变更历史** | 信息变更需审批，保留变更记录 | 无变更审批功能 | 合规风险 |
| **到期提醒** | 证照到期前自动提醒（30天/15天/7天） | 无定时任务 | 证照过期风险 |

## 三、服务人员管理模块 — 差距分析

### 3.1 已有功能（✅）

| 功能 | 实现位置 |
|------|---------|
| 实名认证 | `t_staff.idCard` + 身份证 OCR（预留） |
| 健康档案 | `t_staff_qualification.qualificationType=1`（健康证） |
| 专业证书 | `t_staff_qualification` 表（多类型支持） |
| 技能评估 | `Staff.serviceTypes` 字段 |
| 服务区域 | `Staff.address` 字段 |
| 工时记录 | `t_staff_work_record` 表（checkIn/checkOut） |
| 紧急联系人 | `Staff.emergencyContact` / `emergencyPhone` |
| 服务评分 | `t_staff_score` 表 |

**已有实体**：`t_staff`, `t_staff_qualification`, `t_staff_schedule`, `t_staff_work_record`, `t_staff_score`（5张表）

### 3.2 缺失功能（❌）

| 功能 | 需求描述 | 当前状态 | 影响 |
|------|---------|---------|------|
| **薪资管理** | 基础工资+绩效奖金+加班费自动核算 | 无薪资相关实体 | 无法发薪 |
| **工时统计** | 自动统计实际工时，按月查询 | 仅有 work_record，无统计报表 | 无法结算 |
| **绩效考核** | 5维度考核（满意度/质量/效率/出勤/规范） | 无 t_staff_assessment | 无法评级 |
| **培训管理** | 培训计划/记录/考核/证书 | 无 training 相关表 | 能力提升无据 |
| **请假管理** | 请假申请/审批/统计 | 无 leave 表 | 无法批假 |
| **离职管理** | 离职申请/审批/黑名单 | 仅 leaveDate 字段，无流程 | 无法正规离职 |
| **保险保障** | 平台购买意外保险，保单查询 | 无保险状态字段 | 风险无保障 |
| **排班模板** | 排班冲突检测 + 模板复制 | 无冲突检测逻辑 | 排班可能冲突 |
| **资质审核** | 资质证书需审核（待审核/已通过/已拒绝） | StaffQualification 无 auditStatus | 资质真实性无保证 |

## 五、满意度调查模块 — 差距分析

### 5.1 已有功能（✅）

| 功能 | 实现位置 |
|------|---------|
| 自动触发评价 | 服务完成后自动创建评价记录 |
| 推送渠道 | Token 链接（支持微信/短信） |
| 填写入口 | `GET /api/evaluations/survey/{token}` |
| 服务态度评分 | attitudeScore（1-5分） |
| 专业程度评分 | qualityScore（1-5分） |
| 准时性评分 | efficiencyScore（1-5分） |
| 综合评分 | overallScore（1-5星） |
| 文字反馈 | 直接写 overallScore 字段 |
| 问题追踪 | 评价关联 orderId |
| 改进通知 | 整改流程联动 |
| 积分规则 | evaluation_tags 字段（预留） |

**已有实体**：`t_service_evaluation`（含 environmentScore/professionalScore/tags/images 等扩展字段）, `customer_feedback`

### 5.2 缺失功能（❌）

| 功能 | 需求描述 | 当前状态 | 影响 |
|------|---------|---------|------|
| **评价审核** | 恶意/虚假评价可隐藏或删除 | 无 auditStatus | 差评无法处理 |
| **评价修改（24h）** | 客户24h内可修改评价 | 无 PUT 修改接口 | 评价准确性无保障 |
| **评价补填（7天）** | 超期未评自动4星 | 无定时任务 | 评价率低 |
| **评价置顶** | 优秀评价可置顶展示 | 无置顶字段 | 好评价无法突出 |
| **评价举报** | 客户可举报不合理评价 | 无举报机制 | 恶意评价难处理 |
| **回访评价** | 定期回访收集深度反馈 | 无回访接口 | 深度反馈缺失 |
| **积分奖励** | 参与评价得积分（首次100分） | 无积分账户 | 激励不足 |
| **短信评价** | 短信回复方式提交评价 | 无短信集成 | 渠道单一 |
| **评价率统计** | 评价率=已评订单/总订单 | 统计接口无此字段 | 数据不完整 |
| **排名变化** | 服务商/人员评分排名实时变化 | rankChange 字段存在但未计算 | 激励效果弱 |

## 六、开发方案与步骤

### 6.1 开发优先级矩阵

| 优先级 | 功能 | 开发工时预估 | 风险 |
|--------|------|------------|------|
| **P0** | 服务商审核流程 + 信用评分体系 | 3-4天 | 中（需确认审核流程） |
| **P0** | 服务人员薪资管理 + 工时统计 | 3天 | 低（逻辑清晰） |
| **P0** | 满意度评价审核 + 补填机制 | 2天 | 低 |
| **P1** | 服务商绩效考核 + 退出管理 | 2-3天 | 中（需确认考核维度） |
| **P1** | 服务人员绩效考核 + 请假管理 | 2-3天 | 中 |
| **P1** | 质检标准配置 + 信用联动 | 2天 | 中 |
| **P1** | 客户积分体系 + 礼品兑换 | 3天 | 中（需确认礼品池） |
| **P2** | 培训管理 + 保险管理 | 2天 | 低 |
| **P2** | 优秀案例库 + 评价置顶 | 1-2天 | 低 |
| **P2** | 排班模板 + 冲突检测 | 1-2天 | 中 |

### 6.2 阶段一：P0 核心功能（预计 8-9 天）

#### Step 1：服务商审核流程（3天）

```
Day 1:
- 新增 t_provider_audit 表
- Provider 实体增加 auditStatus 字段
- 新增 ProviderAuditController
- 实现审核接口：POST /api/providers/{id}/audit

Day 2:
- ProviderService 增加审核逻辑
- 审核状态变更后自动创建管理员账号
- 前端：服务商入驻表单 + 审核状态展示

Day 3:
- 服务商信用评分计算（5日/30日/累计平均分公式）
- 新增 t_provider_credit 表
- 每日定时任务计算信用分
- 信用分低于阈值时触发预警
```

#### Step 2：服务人员薪资管理（3天）

```
Day 4:
- 新增 t_staff_salary 表
- StaffSalary 实体 + Service
- 工时统计接口：GET /api/staff/work-hours-summary?month=2026-05

Day 5:
- 薪资核算逻辑：
  月薪资 = 基础工资 + (实际工时 × 单价) + 加班费 - 扣款
- 月度薪资报表接口
- 前端：服务人员薪资查询页面

Day 6:
- 薪资确认流程（员工确认→财务复核→发放）
- 薪资明细导出（Excel）
```

#### Step 3：满意度评价增强（2天）

```
Day 7:
```

## 七、风险估计

### 7.1 技术风险

| 风险 | 等级 | 说明 | 缓解措施 |
|------|------|------|---------|
| 信用评分公式争议 | 高 | 综合评分公式中权重参数（0.3/0.4/0.3）可能需要业务方确认 | 需求阶段与业务方确认参数 |
| 薪资计算规则复杂 | 中 | 不同服务商可能有不同定价规则 | 设计可配置的薪资规则引擎 |
| 评价补填影响历史数据 | 中 | 补填后原"未评价"数据被改写 | 补填记录单独存储，不覆盖原状态 |
| 定时任务分布式执行 | 中 | 多实例部署时定时任务可能重复执行 | 使用分布式锁（Redis）或单节点部署定时任务 |
| 大量定时任务影响性能 | 中 | 每小时计算信用分可能影响DB | 改为每日凌晨低峰期执行 |

## 八、待确认事项（业务方）

> 以下事项需与业务方确认后，方可进入开发：

1. **综合评分公式参数**：`(5日平均分×0.3)+(30日平均分×0.4)+(累计平均分×0.3)-投诉扣分`，投诉扣分权重是多少？
2. **服务商准入评分阈值**：评分低于多少分触发退出/暂停接单？
3. **服务人员保险方案**：保险由平台统一购买还是服务商自购？保费谁承担？
4. **积分礼品池**：礼品范围和兑换比例？
5. **审核流程**：注册审核需要几级？区县级/市级/省级？
6. **薪资结构**：基础工资/绩效奖金/加班费的具体计算规则？
7. **服务商退出触发条件**：连续3个月评分低于阈值，还是单月低于阈值即触发？
