## 一、修订说明

### 1.1 服务人员关系定位调整

**重要修正**：服务人员与平台之间是**灵活短期的劳务关系**，不是入职员工。

| 原需求假设 | 实际情况 | 影响 |
|-----------|---------|------|
| 入职员工管理 | 灵活用工，单次结算 | 去掉：薪资管理/月度考核/请假/离职/排班模板 |
| 雇佣社保 | 无社保，无雇佣合同 | 去掉：保险保障/员工保险 |
| 长期排班 | 按需调度，随用随派 | 去掉：排班模板/排班冲突检测 |

### 1.2 接口预留策略

涉及外部交互（工商核验/短信通道/微信通知）的功能，**预留接口，暂不实现**，避免阻塞核心开发。

### 6.2 开发步骤（按优先级）

#### P0-1：服务商审核流程（Day 1-2）

```
实体修改：t_provider 增加 audit_status 字段
新增表：t_provider_audit

新增接口：
- POST /api/providers/register        ← 服务商注册（自动 audit_status=待审核）
- PUT  /api/providers/{id}/audit      ← 管理员审核（通过/拒绝/补充材料）
- GET  /api/providers/audit-list      ← 审核列表查询
- GET  /api/providers/{id}/audit-history  ← 审核历史

Service逻辑：
- 注册后状态=待审核，不允许派单
- 审核通过后状态=已通过，允许派单
- 审核拒绝后可重新提交
```

#### P0-2：服务商信用评分体系（Day 3）

```
新增表：t_provider_credit

新增接口：
- GET  /api/providers/{id}/credit     ← 查询信用分
- GET  /api/providers/credit-list     ← 信用分列表（排序）

计算公式（参考需求）：
综合评分 = (5日平均分×0.3) + (30日平均分×0.4) + (累计平均分×0.3) - 投诉扣分

定时任务（每日凌晨执行）：
- 从 t_service_evaluation 聚合各服务商的评分
- 计算 5日/30日/累计平均分
- 更新 t_provider_credit 表
- 低于阈值（如3.0分）触发预警
```

#### P0-3：服务人员服务费结算（Day 4-5）

```
新增表：t_staff_settlement

新增接口：
- POST /api/staff/settlements        ← 结算单次服务费（派单完成后）
- GET  /api/staff/settlements        ← 服务人员结算记录
- GET  /api/staff/settlements/{id}   ← 结算明细
- PUT  /api/staff/settlements/{id}/confirm  ← 服务人员确认收款

计算逻辑：
- 结算触发时机：派单状态 = COMPLETED 或 SERVICE_FINISHED
- 服务费 = 实际服务时长（小时）× 单价
- 单价来源：t_provider_service_type.service_price
- 实际时长：t_staff_work_record.work_duration
```

#### P0-4：服务人员评分汇总（Day 6）

```
新增表：t_staff_rating_summary

新增接口：
- GET  /api/staff/{id}/rating-summary  ← 服务人员查看自己评分
- GET  /api/staff/rating-list          ← 管理员查看所有人员评分排名

数据来源：t_service_evaluation（按 staff_id 聚合）
```

#### P0-5：满意度评价审核（Day 7）

```
实体修改：t_service_evaluation 增加 audit_status 字段
新增表：t_evaluation_audit

新增接口：
- GET  /api/evaluations/audit-list    ← 待审核评价列表
- POST /api/evaluations/{id}/audit    ← 审核（隐藏/删除/置顶）
- GET  /api/evaluations/{id}/audit-history  ← 审核历史
```

#### P0-6：证照到期提醒（Day 8）

```
新增表：t_staff_cert_remind

定时任务（每日检查）：
- 扫描 t_staff_qualification 的 expire_date
- 距到期 ≤ 30天 → 插入提醒记录
- 距到期 ≤ 15天 → 再次提醒
- 距到期 ≤ 7天 → 最后提醒
- 到期后 → 通知服务商该人员资质已过期，暂停派单资格
```

#### P1-1~P1-4：扩展功能（Day 9-13）

```
P1-1: 服务商低分自动暂停接单
  - 信用分低于阈值（如3.0）→ provider.status = 暂停接单
  - 恢复：信用分连续30天高于阈值 → 自动恢复

P1-2: 质检标准配置 + 质检与信用联动
  - t_quality_standard 可配置每类服务的检查项
  - 质检不合格（need_rectify=1）→ 扣减服务商信用分

P1-3: 评价补填定时任务
  - 扫描 7天前仍未评价的订单
  - 自动补填：5项评分=4分，评价来源标记"系统补填"

P1-4: 优秀案例库
  - 评价满分（5分）且有文字反馈 → 自动推荐
  - 管理员确认发布 → 入库展示
```

#### P2-1~P2-4：完善功能（Day 14-17）

```
P2-1: 评价置顶
  - t_evaluation_top 表存储置顶记录
  - 查询时优先展示置顶评价

P2-2: 退出管理
  - 服务商提交退出申请 → t_provider_exit
  - 停止派单 → 处理进行中订单 → 数据归档

P2-3: 绩效考核
  - 月度统计：服务量/评分/投诉率
  - 不做雇佣式考核，只做业务数据统计

P2-4~P2-7: 外部接口预留
  - 身份证OCR：接口定义 + 异常日志，暂不接入
  - 定位/求助：接口定义 + 异常日志，暂不接入
  - 短信评价：接口定义 + 异常日志，暂不接入
```
