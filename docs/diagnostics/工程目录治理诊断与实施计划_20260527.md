# 工程目录治理诊断与实施计划

> 诊断日期：2026-05-27  
> 范围：父仓库、`elderly-care-server`、`dingfeng-work`、文档、SQL 和测试产物。

## 结论

当前仓库功能开发持续推进，但工程资产已经出现三个层次的混杂：

1. 可执行源码、生产部署脚本与阶段性报告混放在根目录。
2. 测试/浏览器运行产物和文档生成器编译输出曾被纳入版本跟踪。
3. SQL、前端仓库归属和历史媒体项目的权威来源不够明确。

这些问题会增加接手成本，也会让部署和问题定位依赖个人记忆。

## 已确认问题

| 等级 | 问题 | 证据 | 处置 |
| --- | --- | --- | --- |
| P1 | 浏览器过程文件被跟踪 | `.playwright-cli/` 已跟踪 185 个运行文件 | 本轮退库并永久忽略 |
| P1 | 文档构建输出被跟踪 | `docs/推介资料/bin/`、`obj/` 含 .NET 输出 | 本轮退库并永久忽略 |
| P1 | SQL 来源分裂 | 根 `sql/`、后端 `sql/production/`、`src/main/resources/sql/production/` 并存 | 本轮声明规则，后续核对合并 |
| P2 | 历史报告散落根目录 | 测试报告、审查报告、清理报告和截图位于入口层 | 本轮按类型归档 |
| P2 | 测试数据 SQL 混在根目录 | `insert_*_test_data.sql`、`clear_all_test_data.sql` | 本轮收拢到 `sql/test-data/` |
| P2 | 前端仓为 gitlink 但缺失 `.gitmodules` 映射 | 父仓索引中 `dingfeng-work` 为 `160000`，`git submodule status` 报错 | 需决定单仓或正式子模块后处理 |
| P2 | 媒体项目处于删除状态 | `video-intro` 和 `video-series/` 已有大批删除 | 视为既有工作，本轮不干预 |
| P0 | 部署凭据曾以明文纳入仓库 | 部署脚本、旧代理说明和部署文档包含服务器密码 | 本轮改用环境变量；必须轮换已暴露凭据 |

## 本轮实施

1. 整理根 `.gitignore`，覆盖浏览器测试产物、文档编译输出与本地状态。
2. 将已跟踪的 `.playwright-cli/`、`docs/.DS_Store`、推介资料 `bin/obj`
   从版本管理中移除；本地生成文件可保留供个人使用。
3. 新增 `docs/README.md` 作为文档导航和归档规则。
4. 将根目录历史测试/审查/运维报告及证据截图迁至 `docs/reports/`。
5. 将阶段变更记录迁至 `docs/changelogs/`。
6. 将本地测试数据脚本迁至 `sql/test-data/` 并标注用途。
7. 清除已跟踪部署文件中的明文服务器密码，脚本统一读取 `DEPLOY_PASSWORD`。

## 后续步骤

### 第二批：数据库脚本统一

1. 对比两个 `production/00_schema.sql` 与现网 schema。
2. 选定唯一生产初始化入口，将增量补丁按执行顺序编号。
3. 将零散资源 SQL 分为 `migrations/`、`seed/` 与 `archive/`。
4. 修正 `README.md`、部署文档和 `HANDOFF.md` 的执行路径。

### 第三批：仓库边界确定

1. 决定 `dingfeng-work` 是正式 Git 子模块还是应纳入单仓管理。
2. 如保留子模块，补充 `.gitmodules` 并固定远端/分支规范。
3. 明确 `video-intro` 与视频素材是否继续归属于业务仓库。

### 第四批：业务整改

与工程治理并行处理已诊断的上传、权限、Base64 大字段和页面按需加载问题，
避免目录更干净但运行性能与数据安全仍不可控。
