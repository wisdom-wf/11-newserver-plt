## 已诊断但尚未实施的业务问题

### P0：上传与文件操作越权

- 人员头像、人员资质写接口未校验服务商/人员归属：
  `StaffController.updateStaff()` 与资质增改删接口。
- 老人照片更新接口缺少数据归属校验：`ElderController.updateElder()`。
- 服务商证照删除遗漏归属校验：`ProviderController.deleteCertificate()`。
- 服务日志新增/修改照片可携带任意 `providerId/staffId/elderId`，并继续进入质检链路。

实施顺序：先补后端归属断言与 API 权限回归测试，再调整前端交互。

### P1：上传交互与数据模型失真

- 老人详情“健康档案照片”仅修改前端状态并提示成功，没有持久化字段/API。
- 人员资质弹窗提供“覆盖/添加”选择，但两条路径实际都执行追加。
- 图片/证照主要通过 Base64 写业务表，缺少服务端大小/MIME 校验、上传进度和失败恢复。
- 服务商证照仅接受图片，不支持 PDF，删除操作缺少确认。

实施顺序：先停用伪成功 UI 或补真实附件模型，再设计统一文件上传 API/存储引用。

### P1：页面加载与懒加载不足

- 健康档案入口用 `pageSize: 1000` 加载老人列表，列表 SQL 同时返回 `photo_url` Base64。
- 选择老人立即发起详情、测量、统计、报告、建议、设备等全部请求，没有 Tab 级按需加载。
- 快速切换老人缺乏请求取消或版本校验，旧请求可能覆盖新选择的数据。
- 服务日志列表 Service 使用 `BaseMapper.selectPage()`，可能将照片大字段随列表返回；已有摘要 VO 未用于该列表。
- `LazyImage` 只能延迟解码，无法避免 Base64 已在 JSON 中传输的成本。

实施顺序：先瘦身列表接口响应，再做 Tab 延迟加载、请求竞态保护和视觉 loading。
