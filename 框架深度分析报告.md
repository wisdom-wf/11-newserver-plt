# SoybeanAdmin 框架深度分析报告

## 一、SoybeanAdmin 框架核心能力分析

### 1.1 请求层设计 (`@sa/axios`)

框架提供两种请求实例创建方式：

| 方法 | 返回类型 | 适用场景 |
|------|----------|----------|
| `createRequest` | 直接返回 `T` | 简单场景，框架处理错误 |
| `createFlatRequest` | `{ data, error, response }` | 需要自行处理错误的复杂场景 |

**`createFlatRequest` 核心设计**：
```typescript
// 返回格式
{ data: T, error: AxiosError | null, response: AxiosResponse }

// 错误处理机制：
// 1. isBackendSuccess() - 判断业务错误 (code === '0000')
// 2. onBackendFail() - 处理登出/Token过期等
// 3. onError() - 请求级别错误处理
// 4. 抛出 Error 被 catch 捕获后返回 { data: null, error }
```

### 1.2 表格 Hook 设计 (`@sa/hooks`)

**`useNaivePaginatedTable` 核心设计**：
- 分页参数格式：`{ page, pageSize }`
- 内置 `defaultTransform` 处理两种后端分页格式：
  - `IPage` 格式：`{ current, size, records, total }`
  - `PageResult` 格式：`{ page, pageSize, records, total }`
- `apiFn` 接收分页参数，返回 API 调用

**分页参数映射流程**：
```
pagination.page → apiFn({ page, pageSize }) → params.page
                                          → params.pageSize
```

### 1.3 状态管理模式

**Pinia Store 设计** (`useAuthStore`)：
- `token` - 认证令牌
- `userInfo` - 用户信息（含 `buttons` 权限数组）
- `isLogin` - 登录状态计算属性

**权限检查模式**：
```typescript
// 通过 useAuth hook
const { hasAuth } = useAuth();
hasAuth('staff:list:edit'); // 检查按钮权限
```

## 三、根因分析

### 3.1 框架能力 vs 实际使用

| 框架能力 | 项目使用情况 | 评估 |
|----------|--------------|------|
| `createFlatRequest` 错误处理 | ✅ 正确使用 | 符合预期 |
| `useNaivePaginatedTable` 分页 | ⚠️ 基本正确，参数名需确认 | 存在风险 |
| `useAuth` 权限检查 | ✅ 正确使用 | 符合预期 |
| `useTableOperate` 行操作 | ✅ 正确使用 | 符合预期 |
| `defaultTransform` 数据转换 | ⚠️ 需验证后端返回格式 | 潜在问题 |
| PersonCard 组件 | ⚠️ 字段映射存在不一致 | 需要优化 |

---

### 3.2 问题分类

| 类别 | 问题 | 根因 |
|------|------|------|
| **框架限制** | `defaultTransform` 依赖固定的分页字段名 | 后端返回字段名不一致时无法自动映射 |
| **业务需求不匹配** | PersonCard 需要同时支持员工和老人两种实体 | 组件设计时未考虑通用性 |
| **使用不当** | 部分字段名存在多种变体（`serviceTypes`/`serviceTypeNames`） | 前后端字段命名规范不统一 |
| **框架能力未利用** | 没有使用框架提供的数据转换层 | 直接操作原始数据 |

## 四、优化建议

### 4.1 分页参数标准化

**问题**：前后端分页参数名可能不一致

**建议**：在 API 层统一转换

```typescript
// src/service/api/staff.ts
export function fetchGetStaffList(params?: Api.Staff.StaffQuery & Api.Common.PaginatingQueryParams) {
  return request<Api.Common.PaginatingQueryRecord<Api.Staff.Staff>>({
    url: '/api/staff',
    method: 'get',
    params: {
      ...params,
      // 统一转换：前端使用 page/pageSize，后端可能是 current/size
      current: params?.page,
      size: params?.pageSize
    }
  });
}
```

**或者**：让 `defaultTransform` 支持更多后端格式

### 4.2 统一字段映射层

**问题**：字段名不一致导致代码中大量 `||` 兼容处理

**建议**：在 API 类型定义层统一

```typescript
// src/typings/api/staff.d.ts
interface Staff {
  staffId: string;
  staffName: string;
  // 统一字段名，避免多种变体
  serviceTypesText: string;  // 原始字段 serviceTypes 的格式化版本
}
```

### 4.3 PersonCard 组件优化

**问题**：组件与员工实体强耦合，无法用于老人

**建议 1**：扩展 props 支持不同实体

```typescript
// props 增加泛型支持
interface Props<T> {
  entity: T;
  nameField: keyof T;
  subtitleField?: keyof T;
  photoField: keyof T;
}
```

**建议 2**：创建两个版本
- `StaffCard` - 服务人员专用
- `ElderCard` - 老人专用（可复用到 `elder/index.vue`）

### 4.4 错误处理标准化

**问题**：`staff/index.vue` 等页面自行处理错误，可能遗漏

**建议**：统一错误处理模板

```typescript
// hooks/common/error.ts
export function useErrorHandler() {
  const message = useMessage();
  
  return {
    handleError(error: any, fallbackMsg: string) {
      const msg = error?.message || error?.response?.data?.message || fallbackMsg;
      message.error(msg);
    },
    handleSuccess(msg: string) {
      message.success(msg);
    }
  };
}
```

### 4.5 健康照片存储格式统一

**问题**：`elder/index.vue` 需要兼容多种格式

**建议**：在后端统一返回格式，或在前端增加标准化方法

```typescript
// utils/photo.ts
export function normalizePhotos(photos: string | string[] | undefined): string[] {
  if (!photos) return [];
  if (Array.isArray(photos)) return photos;
  if (typeof photos === 'string') {
    // 尝试 JSON 解析
    try {
      return JSON.parse(photos);
    } catch {
      // 逗号分隔
      return photos.split(',').filter(Boolean);
    }
  }
  return [];
}
```

## 五、结论

### 5.1 总体评估

| 方面 | 评分 | 说明 |
|------|------|------|
| 框架理解 | ⭐⭐⭐⭐ | 核心能力基本掌握 |
| 错误处理 | ⭐⭐⭐⭐⭐ | 使用正确 |
| 分页使用 | ⭐⭐⭐⭐ | 基本正确，有优化空间 |
| 组件复用 | ⭐⭐⭐ | PersonCard 需优化以支持多实体 |
| 字段映射 | ⭐⭐⭐ | 存在不一致，需统一规范 |

### 5.2 主要风险

1. **后端分页字段名**：`current/size` vs `page/pageSize` 需确认
2. **PersonCard 复用性**：无法直接用于老人档案
3. **字段名不一致**：多处使用 `||` 兼容处理，长期维护成本高

### 5.3 优先优化项

1. **高优先级**：统一 API 层分页参数映射
2. **高优先级**：统一前后端字段命名规范
3. **中优先级**：PersonCard 组件泛化或创建 ElderCard
4. **低优先级**：错误处理标准化
